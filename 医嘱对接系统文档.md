# 医嘱对接系统文档

## 一、项目概述

### 1.1 项目背景
医嘱对接系统用于实现医院信息系统（HIS）与电子病历系统（EMR）、药品管理系统、检验系统等之间的医嘱数据交互，确保医嘱信息在各系统间准确、及时地传递。

### 1.2 项目目标
- 实现医嘱信息的标准化传输
- 支持多种医嘱类型的对接
- 确保数据的安全性和完整性
- 提供完整的对接监控和日志记录

## 二、系统架构

### 2.1 整体架构
```
┌─────────────┐
│   HIS 系统   │
└──────┬──────┘
       │ 医嘱接口
       ▼
┌─────────────┐
│  医嘱对接系统 │
└──────┬──────┘
       │ 数据分发
       ├─────────┬─────────┬─────────┐
       ▼         ▼         ▼         ▼
┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐
│  EMR系统 │ │ 药品系统 │ │ 检验系统 │ │ 手术系统 │
└─────────┘ └─────────┘ └─────────┘ └─────────┘
```

### 2.2 技术选型
- **后端框架**: Spring Boot / .NET Core
- **数据库**: MySQL / PostgreSQL
- **消息队列**: RabbitMQ / Kafka
- **接口协议**: RESTful API / HL7 FHIR
- **数据格式**: JSON / XML
- **日志系统**: ELK Stack

## 三、接口设计

### 3.1 医嘱创建接口

#### 接口地址
`POST /api/orders/create`

#### 请求参数
```json
{
  "orderId": "ORD2025031300001",
  "patientId": "PAT001",
  "visitId": "VISIT001",
  "doctorId": "DOC001",
  "orderType": "01", // 01:药品医嘱 02:检查医嘱 03:检验医嘱 04:治疗医嘱
  "orderCode": "药品编码/检查编码",
  "orderName": "药品名称/检查名称",
  "specification": "规格",
  "dosage": "剂量",
  "frequency": "频次",
  "route": "给药途径",
  "startDate": "2025-03-13T09:00:00",
  "endDate": "2025-03-15T09:00:00",
  "priority": "01", // 01:紧急 02:常规 03:长期
  "remark": "备注信息"
}
```

#### 响应参数
```json
{
  "code": "200",
  "message": "医嘱创建成功",
  "data": {
    "orderId": "ORD2025031300001",
    "createTime": "2025-03-13T09:00:00",
    "status": "01" // 01:待执行 02:执行中 03:已完成 04:已停止
  }
}
```

### 3.2 医嘱查询接口

#### 接口地址
`GET /api/orders/query`

#### 请求参数
| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| patientId | String | 是 | 患者ID |
| visitId | String | 否 | 就诊ID |
| startDate | Date | 否 | 开始日期 |
| endDate | Date | 否 | 结束日期 |
| status | String | 否 | 医嘱状态 |

#### 响应参数
```json
{
  "code": "200",
  "message": "查询成功",
  "data": [
    {
      "orderId": "ORD2025031300001",
      "patientId": "PAT001",
      "orderName": "阿莫西林胶囊",
      "orderType": "01",
      "status": "01",
      "createTime": "2025-03-13T09:00:00"
    }
  ]
}
```

### 3.3 医嘱执行接口

#### 接口地址
`POST /api/orders/execute`

#### 请求参数
```json
{
  "orderId": "ORD2025031300001",
  "executorId": "NUR001",
  "executeTime": "2025-03-13T10:00:00",
  "executeResult": "01", // 01:已执行 02:拒绝执行
  "remark": "执行备注"
}
```

### 3.4 医嘱作废接口

#### 接口地址
`POST /api/orders/cancel`

#### 请求参数
```json
{
  "orderId": "ORD2025031300001",
  "operatorId": "DOC001",
  "cancelReason": "患者转科"
}
```

## 四、数据字典

### 4.1 医嘱类型 (orderType)
| 代码 | 名称 | 说明 |
|------|------|------|
| 01 | 药品医嘱 | 药品相关医嘱 |
| 02 | 检查医嘱 | 影像检查医嘱 |
| 03 | 检验医嘱 | 实验室检验医嘱 |
| 04 | 治疗医嘱 | 治疗项目医嘱 |
| 05 | 手术医嘱 | 手术相关医嘱 |
| 06 | 护理医嘱 | 护理相关医嘱 |

### 4.2 医嘱状态 (status)
| 代码 | 名称 | 说明 |
|------|------|------|
| 01 | 待执行 | 医嘱已创建，等待执行 |
| 02 | 执行中 | 医嘱正在执行 |
| 03 | 已完成 | 医嘱已完成 |
| 04 | 已停止 | 医嘱已停止 |
| 05 | 已作废 | 医嘱已作废 |

### 4.3 优先级 (priority)
| 代码 | 名称 | 说明 |
|------|------|------|
| 01 | 紧急 | 立即执行 |
| 02 | 常规 | 按常规流程执行 |
| 03 | 长期 | 长期医嘱 |

## 五、异常处理

### 5.1 错误码定义
| 错误码 | 说明 |
|--------|------|
| 200 | 成功 |
| 400 | 请求参数错误 |
| 401 | 未授权 |
| 403 | 权限不足 |
| 404 | 资源不存在 |
| 500 | 服务器内部错误 |
| 1001 | 医嘱不存在 |
| 1002 | 医嘱已执行，无法作废 |
| 1003 | 医嘱信息不完整 |

### 5.2 异常响应格式
```json
{
  "code": "1001",
  "message": "医嘱不存在",
  "timestamp": "2025-03-13T10:00:00"
}
```

## 六、安全机制

### 6.1 身份认证
- 采用 JWT Token 进行身份认证
- Token 有效期：2小时
- 支持刷新 Token

### 6.2 数据加密
- 传输层加密：HTTPS
- 敏感字段加密：患者信息、诊断信息使用 AES-256 加密
- 密码存储：BCrypt 加密

### 6.3 访问控制
- 基于角色的访问控制（RBAC）
- 操作日志记录
- 敏感操作二次确认
